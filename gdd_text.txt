Core Gameplay Loop 3.1 The Loop — One Problem Cycle Problem Cycle (repeats 8–12x per match) 1. [Problem Display] Math problem renders in the DOM stem container (via KaTeX). Both players see it simultaneously. Timer starts. 2. [Race to Answer] Both players type/tap an answer in the DOM answer area. Answer is submitted to server via submitAttempt(). Server returns verdict: { correct, winner_id, score_delta, streak }. Client NEVER compares answers locally. 3. [Possession Grant] Winner&apos;s canvas: player sprite animates to ball possession. Loser&apos;s canvas: &apos;Too Slow!&apos; flash. Both canvases sync via bridge. 4. [Shot Phase — 3s] Power bar appears in DOM HUD (right edge). Oscillates 0→100→0%. Player presses SPACE or taps to release. Shot trajectory calculated from power value. 5. [Shot Resolution] Phaser tween animates ball arc on canvas. Basket/miss resolved. Score delta sent to bridge. Both players see score update simultaneously. 6. [Buffer — 0.8s] Brief pause. Correct answer shown to both if either was wrong. Next problem loads. Cycle repeats. 3.2 Match Structure by Mode Mode Structure Failure State Opponent vs-score (primary) 8–12 problems, ~60–90s, live opponent leaderboard 3 lives; timeout = lose 1 life Real player via matchmaking solo Same loop, same problem count 3 lives; timeout = lose 1 life CPU (speed calibrated to tier) flat (diagnostic) Fixed 10-problem batch, no timer pressure None — resetForFlatPhase() None; pure accuracy tracking 3.3 Win / Lose Conditions vs-score WIN: Higher score when match ends (all lives gone or problem count hit). vs-score LOSE: Lower score, or all 3 lives lost before opponent. TIMEOUT per problem: Neither player answers in time limit → no possession, no shot, both lose 1 life. TIEBREAKER: If scores equal at match end, a single sudden-death problem decides — first correct answer and basket wins. flat mode: No win/lose. onBatchDone() fires when all 10 problems are attempted. Accuracy reported. 3.4 Shot Power Mechanic The power bar is the game&apos;s signature mechanic — the physical skill layer that prevents the game from being pure typing speed. Power Zone Range Shot Result Points Green (Perfect) 82%–100% Guaranteed basket 3 pts Yellow (Good) 60%–81% 80% basket probability 2 pts if scored Orange (Low) 35%–59% 40% basket probability 1 pt if scored Red (Air Ball) 0%–34% No basket 0 pts Timeout (no release) N/A Turnover — no shot 0 pts, -1 life Design rationale: a student who solved the problem correctly and quickly still needs to time the shot, creating a second moment of agency that keeps the non-shooting player engaged rather than idle. The bar oscillates at a speed that is learnable but not trivial — Green zone is approximately 0.4 seconds wide at default oscillation speed. 4. Math Content Design 4.1 Topics and Grade Alignment SAG Tier Grades topics (problems.topic rows) Example Problems Pro (default) 3–4 multiplication_facts, division_basic 7 × 8 = ?, 56 ÷ 7 = ? All-Star 5–6 fractions_add, decimals_compare, mixed_ops 3/4 + 1/2 = ?, 0.6 × 5 = ? Varsity (fallback) 2–3 addition_2digit, subtraction_2digit 23 + 47 = ?, 81 − 34 = ? 4.2 Problem Delivery via Bridge Problems are NOT generated client-side. The game calls fetchProblems() from @stratacademy/game-bridge, which returns server-curated problems for the current tier. A refill queue ensures problems are available without latency. In vs-score mode, both players receive the same problem from the server — fairness is enforced server-side. 4.3 Answer Input Primary: Typed numeric input field (DOM, always in focus). Enter or tap submits. Secondary: buildAnswerOptions() from @stratacademy/game-ui generates tap-able multiple-choice alternatives (no local distractor synthesis — options come from server). Fractions: Rendered as KaTeX in the stem. Accepted as &apos;N/D&apos; text input. Wrong answer: Input field shakes (CSS animation). Retry immediately. No life penalty for wrong answers — only for timeout. Server validation: submitAttempt() is always awaited. Score/HP/streak applied from returned verdict only. 4.4 Adaptive Difficulty (Within Match) Problem difficulty seeds ramp with the seed-driven stream from difficulty-pacing.md. If accuracy &gt; 85% at the 4-problem mark: stream advances one difficulty notch within tier. If accuracy &lt; 50% at the 4-problem mark: stream steps down one notch. In vs-score: both players share the same problem stream — symmetric constraint maintained. 5. No canvas elements replicate HUD text. 5.4 Bridge Wiring (Critical Path) Bridge Integration Checklist connect() — on BootScene complete fetchProblems(topic, tier) — on match start; refill queue when &lt; 3 problems remain submitAttempt(answer) — on answer submit; AWAIT before ANY state change verdict.correct — gate to possession grant verdict.score_delta — apply to score display verdict.hp — apply to lives display verdict.streak — apply to streak display createMatchController() — vs-score only; manages lobby, ready states, opponent sync pause() / resume() — on browser tab visibility change terminate() — on GameOverScene mount resetForFlatPhase() — flat mode only; disables failure state onBatchDone() — flat mode only; fires results when all 10 problems done NEVER compare answers client-side. NEVER update score before verdict returns. 5.5 Critical Implementation Notes POT-resize + mipmapFilter: Apply to every Ludo sprite in BootScene. Skipping this causes pixelation — it bites every game. scene.add.dom() gotcha: .node IS the element, not a wrapper. Do not double-wrap. Update loop throw protection: Wrap PlayScene update() in try/catch. An uncaught throw freezes the render loop (black canvas, DOM stuck top-left). dom: { createContainer: true }: Required in Phaser config for DOM overlay to work correctly. frameRate formula: Set anims.create({ frameRate }) = frames / duration from Ludo generation params. OGG audio: Use OGG for all Ludo-generated audio. Best Phaser HTML5 compatibility. 6.
